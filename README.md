---

## ⚠️ 这个 skill 能做什么，不能做什么

**这是扫地机，不是雕刻刀。**

skill 能帮你清除明显的 AI 污染 —— 赋能 / 助力 / 让我们一起 / 璀璨画卷 / 🚀 emoji 列点 / `**重点**` 机械加粗 / 希望对您有帮助 —— 让稿子从「一眼像 AI」变成「不像 AI 了」。

但它**不能**帮你写出好文章。好文章背后是你自己看见了别人没看见的东西,敢下别人不敢下的判断。是那些只有在场才写得出的细节。是一段论述、一段叙事推到底的那口气。这些东西,skill 给不了你。稿子的问题要是「没想法 / 观点平庸 / 细节塑料 / 论述泄气」,跑完的结果往往是**从平庸变成更干净的平庸**。

> 让一份没观点的稿子去掉 AI 腔，得到的是一份没观点的稿子，不是一份好稿子。

### 这个 skill**适合**

- ✅ 想把 AI 生成的初稿（ChatGPT / DeepSeek / Claude / Qwen 出的第一版）改得不那么像 AI 的写作者
- ✅ 写工作邮件、PRD、汇报、技术博客、产品文案、公众号推送的日常场景
- ✅ 需要批量清理稿件里明显 AI 腔的编辑和运营
- ✅ 学生、产品经理、运营、程序员、创业者 —— 想让自己的日常写作别那么 AI 化
- ✅ 中文教学、AI 写作识别研究、文本对比分析等场景

### 这个 skill**不适合**

- ❌ 想用工具替代思考的人 —— 如果你的稿子问题是「没有独到观察 / 观点平庸 / 细节塑料」，skill 救不了
- ❌ 严肃深度写作（《三联生活周刊》/《人物》/《GQ 报道》等特稿级别）—— skill 能帮到底层文字清洁，但真正的写作水位靠人的观察、采访、思考，不是靠扫地机
- ❌ 学术论文写作 —— skill 会误判「进行 + V」「然而 / 此外」「性 / 化 后缀」等学术刚需表达（虽然有语体识别保护，但风险仍在）
- ❌ 法律文书、公文 —— 公文腔是合规要求，不是 AI 腔，skill 对这类文本应几乎不动
- ❌ 需要保留叙事铺垫、控制篇章节奏的写作 —— skill 容易误判"铺垫"（环境描写、人物声口前的停顿、离题式背景段落）为"冗余"删掉；也管不了篇章整体的**势与气**（虎头蛇尾、关键处泄气、论证徘徊不前）。这类写作，skill 最多跑一遍做底层清洁，结构和节奏必须作者自己守住
- ❌ 用来「把 AI 生成的内容洗白到不被检测出来」—— 请别拿这个 skill 绕过学校 / 期刊 / 公司的 AI 检测政策。skill 不是作弊工具，是让**你自己的真实写作更清爽**的工具

### 一句话定位

**适合把你自己的初稿改好，不适合用来伪装你没写过的稿子。**

---

本 skill 受 [`humanizer`](https://github.com/blader/humanizer) 启发(作者 Siqi Chen,MIT 协议,2025)。humanizer 官方定位是语言无关的工具,但那 29 条规则主要针对英文 —— Title Case、em dash、hyphenated pairs 都是英文特有现象。

中文版向 humanizer 学了骨架:三遍工作流、语音校准思路、"个性与灵魂"这一章的写法,以及大约一半的 AI 腔模式(humanizer 同类规则的中文改写)。另一半是中文语境下重新整理的:语体识别前置步骤、汉语语法依据、标点全角、语序去欧化、避免过度修正的五问。#39-#42 四条模式另有出处,来自 yage.ai 的原创文章。

## 安装

### Claude Code

```bash
# 方法一:git clone(推荐)
git clone https://github.com/hzblacksmith/qu-ai-wei.git ~/.claude/skills/qu-ai-wei

# 方法二:手动下载
# 从 GitHub 下载 zip,解压到 ~/.claude/skills/qu-ai-wei/
```

### OpenCode

OpenCode 也会扫描 `~/.claude/skills/`,所以 Claude Code 的安装会自动被识别。或者单独放在 `~/.config/opencode/skills/qu-ai-wei/`。

### Cursor / Windsurf

把 `.cursorrules` 复制到项目根目录：

```bash
# 先 clone(如已装 Claude Code 可跳过)
git clone https://github.com/hzblacksmith/qu-ai-wei.git ~/qu-ai-wei

# 复制到项目根
cp ~/qu-ai-wei/.cursorrules /path/to/your-project/.cursorrules
# Windsurf:同时复制一份为 .windsurfrules
cp ~/qu-ai-wei/.cursorrules /path/to/your-project/.windsurfrules
```

之后在 Cursor / Windsurf 里说「帮我去 AI 味」「改得说人话」即可自动触发。

### Warp

把 `WARP.md` 复制到项目根目录，Warp 的 AI agent 会自动把它当作仓库级系统指令加载；想全局生效可以放 `~/.warp/WARP.md`：

```bash
# 先 clone(如已装 Claude Code 可跳过)
git clone https://github.com/hzblacksmith/qu-ai-wei.git ~/qu-ai-wei

# 项目级
cp ~/qu-ai-wei/WARP.md /path/to/your-project/WARP.md
# 或全局
cp ~/qu-ai-wei/WARP.md ~/.warp/WARP.md
```

之后在 Warp 里说「帮我去 AI 味」即可触发。

### 其他支持自定义指令的模型（ChatGPT / DeepSeek / Kimi / 通义 等）

把 [`SKILL.md`](./SKILL.md) 的正文直接粘到模型的自定义指令或系统提示里，跳过顶部 `---` 包起来的 YAML frontmatter 那段。

## 用法

### 自然语言触发（推荐）

直接用人话让模型处理，skill 会自动触发：

```
帮我去 AI 味:[粘贴中文]
改得说人话:[粘贴中文]
这段中文太 AI 了,润色一下:[粘贴中文]
让它更像人写的:[粘贴中文]
humanize 这段中文:[粘贴中文]
```

### 显式调用（slash 命令）

```
/qu-ai-wei

[粘贴你要改写的中文]
```

### 字形范围

**只处理简体中文。** 繁體输入会提示"当前版本只支持简体中文，建议用 OpenCC 等工具先转简体再回来",不做自动转换。

| 调用 | 行为 |
|---|---|
| `/qu-ai-wei <text>`（简体） | 按 50 条规则 + 语体识别 + 过度消毒反制约束改写 |
| `/qu-ai-wei <text>`（繁體） | 提示用户转简体 |
| `/qu-ai-wei`（无参） | 询问用户粘贴文本；可选做轻量级语音校准 |

### 语音校准（可选）

想让改写贴近你自己的写作风格，可以粘一段你自己写的中文做参考：

```
/qu-ai-wei

以下是我自己的写作样本,用来做风格参考:
[粘贴 2–3 段你的文字]

现在请改写这段:
[粘贴 AI 写的中文]
```

skill 会先从样本里提取一份**5 项轻量清单**（高频词、平均句长、段首词、标点偏好、整体语域），你确认之后再改写。比起完整的风格分析，这种做法更诚实：2–3 段中文能给的信号本来就有限。

## 设计思路

### 与 humanizer 的关系

骨架和方法学借自 humanizer:三遍工作流、语音校准、"个性与灵魂"章节、大约一半的模式概念。剩下的在中文语境里重新整理了一遍。

| 方面 | 英文 humanizer | qu-ai-wei(0.3) |
|---|---|---|
| 目标语言 | 定位语言无关,实际规则针对英文 | 简体中文 |
| 规则数量 | 29 类(参考 Wikipedia "Signs of AI writing") | 51 类 + **顶层 5 条硬约束**（过度消毒反制 + AI 不敢写正向检测 + 事实发明禁令 + 门检强制输出 + 语体降级保护）;约一半是 humanizer 同类规则的中文改写,其余为中文原创(含中文维基《AI生成文的特徵》启发的 4 条,2024-2025 AI 腔新形态 3 条) |
| 规则关系 | 针对英文语法、英文 AI 腔 | 针对中文语法(吕叔湘、朱德熙、汉语语法维基等)、中文 AI 腔 |
| 语体区分 | 不区分 | 9 种语体前置识别(社交 / 自媒体 / 商务 / 书面 / 特稿 / 品牌广告 / 学术 / 公文 / 高考应试作文) |
| 方法学 | 模式识别 + 密度判定 | 同(借鉴自 humanizer) |
| 工作流 | 三遍改写 | 三遍改写 + 过度修正五问自查(反向翻译 / 频率 / 对标 / 话题指代 / 语体匹配) |
| 语音校准 | 完整风格分析 | 轻量化 5 项清单 |

### 核心能力:语体识别

如果把所有文本都往"日常白话"改，会把学术论文的"进行深入分析"改成"好好分析一下"、把公文"依法予以处理"改成"按法律办"。所以 skill 先做**语体识别**,再选规则：

| 语体 | 典型场景 | 去 AI 腔激进度 |
|---|---|---|
| 社交 / 口语 | 微信、朋友圈、豆瓣 | 最激进 |
| 内容 / 自媒体 | 公众号营销号、小红书、短视频 | 激进 |
| 商务 / 职场 | 邮件、汇报、PRD | 中等 |
| 书面 / 一般 | 博客、随笔、科普、时评 | 中等 |
| **叙事非虚构 / 特稿** | 《人物》《三联》《GQ 报道》等深度长读 | 中等偏保守（保留气氛铺垫、排比气势、慢节奏） |
| **品牌广告 / 文案** | Apple 官网、Nike campaign、发布会大字、户外大牌、TVC 旁白 | **特殊档**（严查翻译腔和 AI 商务词，但保护短句、留白、排比、文言感、中英混排 —— 整套规则反着走） |
| 学术 / 科技 | 论文、白皮书 | 保守(保留 `进行 + V`、`性 / 化` 后缀等学术刚需表达) |
| 公文 / 法律 | 法规、合同 | 最保守（仅改客服腔） |
| **高考 / 应试作文** | 高考 / 中考 / 本科应试作文 | **最保守**(仅改客服腔与格式幻觉；排比、引用、成语气势、`进行 + V`、`性 / 化` 后缀、逻辑连接词一律保留 —— 都是评分加分项) |

SKILL.md 里有完整的规则激活矩阵。判不清的时候 skill 会问："这段文字发在朋友圈 / 公众号 / 工作邮件 / 特稿 / 品牌官网 / 学术报告 / 合同 / 高考作文里？"

### 核心检测原则：看密度，不看出现

对仗、排比、四字成语,写中文的人本来就在用,真人写作里到处是。光是用了这些不算 AI 腔；反复堆叠、跟前后内容没关系、换到别的话题上也成立,这才是。单次出现不算,短段(200 字以内)里反复出现才算。

### 避免"过度修正"

这 51 类模式是检测启发式，不是改写模板。去 AI 腔 ≠ 去一切规范化表达。真人常用、共识度高的中文写法（比如"支持 X 等平台""用于 X""基于 X"）本身不是 AI 腔。改写前多问五步：

1. **反向翻译验证**:倒译回英文，是不是像 `runs on` 或 `is compatible with` 的直译？是就改回原文。
2. **频率验证**:这个表达在真人中文写作里常见吗？不常见 + 我临时发明 → 危险。
3. **对标验证**:放进《三联生活周刊》或微信系统提示里，会不会扎眼？会就停下。
4. **话题 / 指代验证**:`这是一个 X` 中的 `这是` 经常承担话题引入功能，不是冗余系动词。只在 `X 是 X` 纯冗余结构才删。
5. **语体匹配验证**:改写后的句子跟原文**语体级别**是否一致？把学术严谨改成日常口语、把公文庄重改成朋友圈调侃，都是失败的改写。

### 关于正面参考

skill 里列了《三联生活周刊》《读者》《中国国家地理》、微信 UX，还有汪曾祺、鲁迅、阿城这些人，作为"正面对照"。v0.4 新增 **Apple 大中华区 / Nike 大中华区**作为**品牌广告语体**的反向对照基准（"这段文字放进 Apple 官网 hero copy / Nike 广告片里会刺眼吗？"）。用法要注意:它们是**诊断时的反面参照**("这句话放进《三联生活周刊》会不会别扭？")，不是**风格模仿目标**("请按三联风格写" / "写个 Apple 风 slogan")。只给 LLM 一个名字让它学风格，出来基本都是刻板仿作。一键风格切换暂不做。

### 中英混排：只保留专有名词，不管空格

**唯一的硬性规则：** 英文专有名词、产品名、人名、缩写，改写时**原样保留**,不翻译、不意译、不替换。"Claude Code" 不能改成"克劳德代码","Transformer" 不能改成"变换器"。这是"信息完整性"的延伸。

**中英、中数之间的半角空格(俗称"盘古之白"),本 skill 不管。** 这东西 AI 和人都在各自变,加或不加根本分不出来,不能作为去 AI 腔的信号。也不是国标 — GB/T 15834-2011 只管标点,W3C CLREQ 说"字距或空白"均可不强制 U+0020,sparanoid 盘古之白原作者自己都说它是 "informal typographic convention"。真要替用户加反而踩坑:人家原文本来不加空格(朋友圈、公众号、小红书都这样),skill 给加上,一股"技术圈腔"就冒出来了。

**规则:原文有空格保留,没空格不加。** 需要统一格式或追求中文排版美学，用 [pangu.js](https://github.com/vinta/pangu.js) 自动处理，或参考 [sparanoid 的中文文案排版指北](https://github.com/sparanoid/chinese-copywriting-guidelines/blob/master/README.zh-Hans.md) —— 那是排版规范，跟本 skill 的"AI 痕迹检测"是**不同目标**（排版管美学、我们管机械感）。

### 标点符号：中文正文一律用全角

中文上下文里，标点符号一律用全角，不跟半角混用。英文训练数据里标点全是半角，模型输出中文时常常直接复用，读起来就有"翻译腔"的视觉信号。对照表、例外清单、典型错误都写在 SKILL.md 的「标点符号」一节。

### 语序：完全中文化，避免英文式倒装

LLM 常把英文句法直接套到中文上。单句读着像中文，一连读就发现别扭，多半是语序带了英式残留。中文的基本习惯：主语前置，定语要短，状语在动词前，条件和原因从句放主句之前。常见病（介词短语前置过长、"作为一个 X,..." 开头、定语堆叠、英文关系从句直译等）和自查四问写在 SKILL.md 的「语序」一节。

### 信息完整性

改写只改表达方式，**不删信息**。原文里每一条具体事实（数字、人名、时间、产品名、关键措辞），终稿里都要能对应上。觉得某处表达冗余，可以压缩；但不能悄悄丢事实。有疑问的时候，保留原信息比追求简洁更重要。

## 51 类模式一览

### A. 内容模式 · Content patterns

| # | 模式 | 原文 | 改后 |
|---|---|---|---|
| 1 | **过度拔高意义** | "标志着...重要里程碑，谱写了新的篇章" | "1989 年成立，负责本地区统计工作" |
| 2 | **空洞的关注/强调句** | "值得一提的是，不可否认..." | 直接陈述 |
| 3 | **时代背景开场套路** | "随着 AI 技术的不断发展..." | 具体事件、具体时间 |
| 4 | **尾巴式空洞总结** | "...体现了...深刻内涵" | 说清楚具体是什么 |
| 5 | **模糊权威归因** | "专家表示、业内人士认为" | 具名人物 + 具体出处 |
| 6 | **公式化"挑战与展望"段** | "尽管...但未来必将更加光明" | 具体难点 + 具体数据 |

### B. 语言模式 · Language patterns

| # | 模式 | 示例 |
|---|---|---|
| 7 | **AI 高频词过载** | 赋能、助力、打造、护航、抓手、闭环、生态、底层逻辑、高质量发展 |
| 8 | **华丽意象词堆砌** | 璀璨、熠熠生辉、画卷、乐章、华章、扬帆起航 |
| 9 | **同义词轮换** | 主角 → 主人公 → 核心人物 → 英雄 |
| 10 | **"不仅...更..."空洞并列** | "不仅是 X,更是 Y"（单用无妨，密集堆叠才是 AI 腔） |
| 11 | **的的不休** | 短句连用三个以上 `的` |
| 12 | **滥用 `进行 + V`** | 进行讨论/分析/研究 → 讨论/分析/研究 |
| 13 | **虚假范围 "从...到..."** | X/Y/Z 不在同一个刻度 |
| 14 | **`性` 后缀堆叠** | 可持续性、创新性、系统性、全面性 |
| 15 | **`化` 后缀堆叠** | 数字化、智能化、多元化、网络化 |
| 16 | **长破折号解释句** | `X ——也就是 Y` 的同位补注（段段都来一个） |
| 17 | **中英术语强行括注** | 算力(compute)、推理(inference) 冗余括注 |
| 18 | **过度数字化列点** | 连贯叙述被硬拆成 `1. 2. 3.` 编号 |
| 19 | **"需要注意的是..."空枢纽** | 不承载信息的提示句 |
| 20 | **"这里有几个要点"式列表预告** | 宣告要做某事，而不是直接做 |

### C. 修辞模式 · Style patterns

| # | 模式 | 示例 |

## 完整示例

**原文（AI 腔很重）：**
> 问得好！下面是关于 AI 编程助手的简要介绍，希望对您有帮助！
>
> 随着人工智能技术的不断发展，AI 辅助编程正以前所未有的方式，赋能千行百业，助力开发者打造高质量、智能化、一体化的产品。这不仅是一场技术革新，更是一次思维变革，它深刻体现了技术对人类生产方式的重塑，彰显了数字化时代的无限可能。
>
> 值得一提的是，业内人士普遍认为，AI 编程工具在多个维度实现了突破：
>
> 1. 🚀 **效率跃升：**代码生成速度显著提升。
> 2. 💡 **质量保障：**通过深度学习，代码质量得到全面优化。
> 3. ✅ **体验升级：**协作体验得到前所未有的改善。
>
> 需要注意的是，尽管面临诸多挑战，但我们有理由相信，在不久的将来，AI 编程必将迎来更加光明的未来。让我们一起，共同见证并书写这个充满无限可能的新篇章！

**自审（初稿之后，再扫一遍）：**

去掉"赋能 / 让我们一起 / emoji 列点"这些显眼 AI 腔只是第一步。再过一遍：

- 每段一个 neat parallel 叠起来，读着就是对仗太工整 —— AI 偏爱这种对称
- 收尾"你基本上...判断"——第二人称 + 抽象动作，活脱脱标语句式
- "两种人"这种 generic label 还是抽象。写一个真朋友的具体场景，比笼统归类有力

**终稿（扫过自审再改一遍）：**
> AI 编程助手在无聊的部分能帮你提速。不是所有部分。架构这事它不行。
>
> 模板代码它写得挺顺 — 配置文件、测试脚手架、重复的重构。它也挺擅长"看起来对"但实际不对。我接受过几次建议，代码跑起来也过了 lint,后来才发现根本不是我要的，因为我没看。
>
> 我见过两种用法。一个朋友把 Copilot 当补全器，每行都要看过才 tab 下去；另一个直接关掉了 —— 他说它老建议他们项目里两年前就弃用的写法。两种我都理解。
>
> 生产力指标不好拿。GitHub 可以说 Copilot 用户"接受了 30% 的建议",但接受不等于正确。没测试的时候，我分不清它哪次真帮了我，哪次是 lint 放过去的假助攻。

详细的三遍工作流（识别 → 初稿 → 自审 → 终稿）和完整改动摘要参见 [SKILL.md](./SKILL.md#完整示例)。

## 给贡献者 / 自己维护的几条提醒

1. **改了 SKILL.md / README 后，把自己动过的散文段落**喂给 skill 自检一遍。规则条目里那些结构化 metadata(问题 / 关键词 / 原文 / 改后 / 语体限定)是**给模型读的骨架，别 humanize** —— 去掉对称反而让模型看不清规则长什么样。**只对解释性散文跑。**
2. **加新规则时检查会不会跟既有规则冲突 / 重叠**。比如"抽象万能动词"跟 #7 AI 高频词、#30 冗余书面化都相邻，要说清楚各自管的是什么。
3. **所有例子都要有"原文 / 改后"对**。单给判断标准没有示范，模型抓不准。
4. **版本号凡升，README 的 Version History 和 SKILL.md frontmatter 同改**。
5. **遇到真人经典文本（金庸 / 王朔 / 汪曾祺 / 真实采访实录等）不要改**。这是 skill 最常见的灾难，在"🛑 第负一步"明确写了门检。

**三方集成测试驱动的护栏加固**。基于 QWEN International (qwen-max-latest) 执行 skill → GLM (glm-4.6) 打分 → 独立 Sonnet 4.6 agent 复核的三方闭环测试，发现**事实发明**是 skill 当前最严重的执行漏洞：在三段不同语体的 AI 腔原文（商务公告 / 小红书裸辞 / LLM 技术综述）上，执行方模型（QWEN）系统性地在终稿里虚构原文没有的具体细节（`快了 3 倍` / `试了两轮内测` / `我们团队` / `妈妈打电话` / `同事晒升职`），并借此通过"AI 不敢写测试"得分。评测方（GLM）完全未识别这种虚构,给出满分。这一版的修改是对这类漏洞的系统性封堵。测试产物保留在 [`tests/integration-run/`](./tests/integration-run/),可作后续 regression。

- **新增顶层硬约束：事实发明明令禁止**。在「⚠️ 信息完整性」章节补「🚫 硬约束：不得在终稿中发明原文没有的具体事实」子节——明确禁止发明具体数字、事件、不存在的主体、金额时间地点、引用原话。"注入人味"的合法路径收窄为一条：**只能从原文已有信息里挑、提炼、具体化,或以明显的作者视角补注**,不允许虚构。遇到原文真没有毛边时,正确做法是告知用户"我不替您编",而不是默默编造。
- **强制门检输出格式**。在「🛑 第负一步」节补「✅ 门检必须显式输出判断声明」—— 无论判断结果如何,都必须在任何改写工作之前输出一行 `【门检】判断：AI 生成文本 / 真人文本（停手） / 不确定 | 证据：[≤2 条具体信号]`。没有这行,用户和评测者无法审计门检是否真的执行了。证据要写原文里的具体词/短语,不要写抽象判断。
- **技术博客语体归档**。在「九种基础语体」节补技术博客 / 工程博客 / AI 科普文的归档判据:短段 < 300 字出现 ≥ 3 个英文技术全拼词 或 出现"中文（English full name）"首次引入格式,按「学术 / 科技」处理。英文白名单扩展,`latency` / `context` / `throughput` / `quantization` / `pruning` / `embedding` / `inference` / `attention` / `token` / `kernel` 等工程全拼术语与 `API / SDK / LLM / RAG` 同级保留英文,#42 不触发。原因:集成测试中 Sample C 的 `context` / `latency` 被 QWEN 当作 #42 翻成"上下文 / 延迟",让有经验的技术读者觉得生硬。

- **新增 `### 人物昵称：中文社区沉淀的"第三形态"` 子节**（在「中英混排：保留专有名词」章节内,跟 `### 体育专有名词` 、 `### 通用技术 / 互联网缩写` 平级）。分三档：体育圈（`黑曼巴 / 约老师 / 大帝 / 闪电侠 / 小皇帝`）、娱乐 / 饭圈（`荷兰弟 / 霉霉 / 大表姐 / 卷福 / 抖森 / 一美 / 法鲨`）、科技圈（`老黄 / 小扎 / 劈柴`）。

- **触发场景区分：** 日常写作 / 体育报道 / 娱乐新闻 / 自媒体保留昵称；正式讣告 / 官方声明 / 学术引用首次展开一次,后文回到昵称（跟 MCP 首次引入同理）。

### v0.5.1 (2026-04-21)

基于中文社区的**本土英文使用惯例**，新增「中英混排」白名单的第一批内容。起因：中文技术圈 / 互联网圈写作里 `MCP / TLDR / BTW / API / SDK / PR` 等词**本来就是英文形态**,AI 机械翻译成"模型上下文协议 / 太长不看 / 顺便说一下"反而刺眼。原有规则 #42 没有明确白名单，容易过度触发把这些词也翻掉。

- **新增 `### 通用技术 / 互联网缩写` 子节**（在「中英混排：保留专有名词」章节内,跟 `### 体育专有名词` 平级）。分四档：产品 / 协议 / 平台名（`Claude Skills / MCP / Copilot`）、通用技术缩写（`API / SDK / CLI / PR / CI/CD / LLM / RAG`）、互联网速记（`TLDR / BTW / FYI / TBH / IMO`）、文档状态标记（`TBD / WIP / TODO / FIXME`）。规则：**一律保留英文原形，不翻译、不括注中文**。
- **更新规则 #42（有中文译法却留着的英文词）**,在例外列表顶部加入主白名单的 cross-reference,并明确"本规则只管普通英文词该翻未翻（`argument` → `论证`），不管中文圈本来就用英文形态的词（`MCP / TLDR / API`）"。
- **更新规则 #17（中英术语强行括注）**,加澄清：独立使用的英文缩写（`MCP / TLDR`）**不是 #17 的情况**,#17 专管「中文词 + 括注英文原文」的冗余并列。典型错改路径被堵住：`用了 MCP` → `用了 MCP（Model Context Protocol）` → 再报 #17 —— 第一步就不该发生。附首次引入的正确格式（解释性括注 ≠ 翻译性括注）。
- 总规则数仍为 51。顶层硬约束仍为 2 条。

### v0.5.0 (2026-04-21)

基于对 Tim Cook 2026 年致 Apple 社区告别信的中英文对照分析，提炼了两条规则。这封信是**CEO 正式公开信**语体里罕见的"不像 AI"范例（*you* 30+ 次 / *users* 2 次、独立成段的超短句 `This is not goodbye.` `Tim.`、主动把身份降格为 *"a person who grew up in a rural place in a different time"*），可作为该语体的反向诊断锚点。

- **新增顶层硬约束：终稿自检·AI 不敢写测试**。原有 `过度消毒反制` 章节只有"前后对比式"自审（改过头没有），缺一个**终稿本体**检测。新测试不依赖原文，直接问终稿：有没有**任何一句**是 AI 不敢写的 —— 自嘲 / 承认不确定 / 冒犯性观点 / 身份降格 / 只有当事人才知道的具体细节？一句都没有 → 太干净，回去加一处。跟 `硬约束`（保留原有毛边）互补，两者都过才算改写合格。
- **新增规则 #51：第二人称泛化代词**。AI 中文直接对读者说话时，默认主语是"用户 / 大家 / 各位 / 朋友们"，真人信件 / 随笔 / 公开信绝大多数用"你"。诊断：直接对话段落里"用户 + 大家 + 各位" ≥ 3 次 且"你 + 你们"= 0 次 → 触发。区分第三方指代（"产品的目标用户"，保留）和对话指代（"大家在使用产品时"，改成"你"）。学术 / 公文 / 高考应试作文不触发。附 Tim Cook 告别信的正反对比示例。
- 总规则数 50 → 51,总语体数仍为 9。顶层硬约束由 1 条（过度消毒反制）扩展为 2 条（过度消毒反制 + AI 不敢写测试）。

### v0.4.0 (2026-04-21)

- **新增第 9 种语体「品牌广告 / 文案」**。之前 8 种语体（社交 / 自媒体 / 商务 / 书面 / 特稿 / 学术 / 公文 / 高考作文）全都覆盖不到 Apple CN 官网、Nike 大中华区 campaign、发布会大字、TVC 旁白、户外大牌这类场景 —— 这些文案**跟其他语体整套规则几乎反着走**：允许短到不能再短的句子（`岂止于大` / `管什么输赢,上`）、允许留白和不完整句、允许文言感、允许中英混排（`iPhone` / `Mac` / `Transformer` 保留英文）、允许排比做节奏设计。之前按 "商务" 或 "自媒体" 语体套规则,会被加回"赋能助力"或塞回"家人们",彻底失真。
- **规则激活矩阵扩展到 9 列**,品牌广告语体对每一类规则（A-I）都有专门判据。核心逻辑：**严查翻译腔和 AI 高频商务词（赋能 / 助力 / 打造 / 闭环）,严防堆意象（璀璨 / 乐章 / 华章）,但保护短句、留白、排比、文言感、中英混排。**
- **新增「品牌广告 vs 自媒体营销号：三条识别线」**。两个语体最容易混,但实际是不同物种 —— 说话的人（企业 brand voice vs KOL 人设）、CTA 形式（`Just Do It / 上` vs `点赞收藏关注`）、信息密度（极度压缩 vs 填充式）完全相反。附四个边界场景的归类（品牌 PR 稿走商务、品牌 SCRM 话术走 #37-E 等）。
- **新增「品牌广告 / 文案语体示例」5 组**（原文/改后）：产品发布 hero copy、campaign 主视觉、节日 campaign、户外大牌短 slogan、产品介绍（中英混排处理）。每组带改动摘要和"品牌文案视角"判据。附硬约束："这种改写只在品牌广告语体里适用,不要套到博客 / 特稿 / 邮件上。"
- **「正面参考模型」章节新增「品牌广告 / 文案 语体锚点」**：Apple CN 经典例（岂止于大 / 让妈,更妈 / 唯一的改变,是一切都变了）+ Nike CN 经典例（管什么输赢,上 / 跑了就懂 / 活出你的伟大）+ 小米 / 李宁 / NB / 支付宝 UX。附四个硬信号检查表（短到不能再短 / 没有赋能助力 / 产品名保留英文 / 标点克制）。
- **#37 自媒体套路规则的语体限定加上"品牌广告不触发"的明确说明**,避免模型把 #37 的"伪深度咨询腔""伪疗愈"诊断误套到真·品牌文案上。
- 总语体数 8 → 9,总规则数仍为 50（无新规则,是新语体对现有规则的重新组织）。
